# 1日目 | シリーズ紹介とガイド: ゼロから構築する納品可能なシステム設計 - AWSを例として

## なぜアーキテクトには哲学的思考が必要なのか?

ChatGPTはコードを書くのを手伝ってくれますが、「このシステムの存在目的は何か」を教えてくれません。これこそが、アーキテクトにとっての哲学的思考の核心的価値です。

私がエンジニアになる前は、ライフガード(高校時代のアルバイト)、水泳コーチ(そう、ライフガードとして短期間働いた後、新しい任務を与えられました)、データアナリスト、山岳ガイド、プライベートシェフ、デジタルマーケティングプランナー、さらにはタロットリーダーまで、様々な仕事をしていました。

多くの異なる仕事内容の中で、私はある文脈をまとめ、すべての仕事とやりたいことは、この文脈、つまり**目的志向**に従って実行できることを発見しました。後にソフトウェアエンジニアリングの世界で、これが「**ドメイン駆動設計(Domain-Driven Design)**」と呼ばれることを学びました。

## システムの有機性と目的性

**システムは、機能の完全な実現を表し、その哲学的意味と有機性を持っています。**

生物の生命の目的がそれ自身を永続させることであるように、すべてのシステムの誕生は特定の目的を達成するためです:

- 検索システムは、膨大なデータから欲しい情報を見つけるため。
- AWS Lambdaは、サーバーを管理せずにコードを実行させるため。
- API Gatewayは、すべてのAPIエントリーポイントを統一的に管理するため。
- S3は単なる「クラウドハードドライブ」ではなく、「任意の拡張、高耐久性、低コストストレージ」というビジネス目的を満たすため。

システム設計は実際には概念的なバイオミミクリーです - 私たちはその摂取(入力)、消化(処理)、変換(計算)、目的達成(出力)を設計します。まるで人体のように:

- 消化器系は、入力データの解析と変換に対応。
- 神経系は、イベント駆動アーキテクチャに対応。
- 免疫系は、セキュリティ保護、災害復旧、監視とアラートに対応。
- 循環器系は、データフロー、メッセージバス、API呼び出しに対応。

ソフトウェアエンジニアとして、私たちはシステムを厳密にその「目的」に従って設計しなければなりません。そうしないと災害が発生します。
発生する可能性がある(そして私の日々の経験では確実に発生した)状況を想像してください:

- 完璧なAPIを構築したが、実際のビジネスと接続できない。
- UIの最適化に時間を費やしたが、基礎となるデータフローがユーザーの核心的ニーズを全く満たしていない。

これは、メインディッシュのない精巧な食事や、正しい地図のない山岳ガイドのようなものです。材料が間違っているのではなく、デザイナーが「目的」を忘れたのです。
ソフトウェアエンジニアとして、私たちはシステムを厳密にその目的に従って設計しなければなりません。そうしないと災害が発生します: 工場の基準を満たさない歯車、美食家を満足させられない食事。材料が間違っているのではなく、デザイナーが間違っているのです。

## ドメイン駆動 vs 振る舞い駆動: アーキテクトの視点の違い

多くの人は、BDD(振る舞い駆動開発)でシステムを考えることに慣れており、ユーザーの操作軌跡に沿って機能を設計します。

これは間違いではありません。特に製品検証と迅速な反復の初期段階では。

しかし、密林が草原から徐々に成長するように、システムの複雑さが増すと、異なる操作シナリオが日光を覆う蔓になり、ビジネスロジックのメンテナンスコストがエントロピー的に増加し、最終的には放棄またはリファクタリングの臨界点に達します。

これが、アーキテクトがDDDの高レベルな視点を習得する必要がある理由です。

**DDDの価値は、より高い視点からシステムアーキテクチャ全体を俯瞰できることです。**

DDDの価値は「コードの階層化」だけではなく、私たちを「目的」に戻すことにあります:
ドメインモデルは現実世界の法則の抽象化です。
境界づけられたコンテキスト(Bounded Context)は「このサブシステムに属するものと属さないもの」を定義するのに役立ちます。
集約(Aggregate)は「もの」の完全な定義であり、単なるデータテーブルではありません。

各「ドメイン」に従ってコンテキストを考え、操作シナリオを設計し、状態変化を収束させると、システムに新しく参加した開発者がルールを素早く把握でき、振る舞いの特殊ケースによって構築された泥沼に陥らないようにできます。

DDDの価値は「コードの階層化」だけではなく、私たちを「目的」に戻すことにあります:

- ドメインモデルは現実世界の法則の抽象化です。
- 境界づけられたコンテキストは「このサブシステムに属するものと属さないもの」を定義するのに役立ちます。
- 集約は「もの」の完全な定義であり、単なるデータテーブルではありません。

これは、AWSアーキテクトにとって特に重要です。なぜなら、クラウドサービスのオプションが多すぎるからです。ドメインロジックのガイダンスがなければ、サービスリストで簡単に迷子になります。

## AI時代におけるアーキテクトの価値

特に今、計算能力の爆発により、AIが手動コーディングをはるかに超える速度でコードを生成できるようになった時代では、**ドメイン知識の深さとビジネスロジックの把握が真の競争障壁になっています。** ビジネスドメインと目的を理解し、それらをアーキテクチャと設計に翻訳する能力が、最も重要で最も必要とされる能力になっています。

これが、私が30日間であなたと共有したい核心です: AWSツールの使い方だけでなく、ドメイン駆動思考を使ってクラウドアーキテクチャを設計する方法です。

AIはコードの構文の詳細を完成させるのを手伝ってくれますが、以下の質問には答えられません:

「このサービスはモノリシックアーキテクチャに配置すべきか、マイクロサービスに配置すべきか?」
「データベースはパーティション化すべきか? ビジネスの観点からなぜパーティション化する必要があるのか?」
「国境を越えた展開は単純な技術的選択肢か、ビジネス戦略の要件か?」

これらの質問の背後には哲学的思考があります: 「なぜ」を問い、単に「どうやって」を問うのではありません。

## アーキテクトの日々のジレンマと考え方の転換

多くの人は、アーキテクトの仕事は「アーキテクチャ図を描く」ことや「AWSサービスを選択する」ことだと考えています。実際、これは氷山の一角に過ぎません。

現実世界では、アーキテクトは以下のジレンマに直面します:

1. 要件は常に曖昧

- ビジネス部門はよく言います:「Uberのようなものが必要です。」
- この文は何を意味するのか? 配車? 支払い? リアルタイムロケーション? ドライバー評価?
- アーキテクトは、曖昧なビジョンを明確なシステム境界に分解できなければなりません。

2. 技術は常に変化している

- 今日AWSが新しいサービスをリリースし、明日Google CloudまたはAzureがより安価な代替案を提供するかもしれません。
- アーキテクトは「トレンドを追いかける」罠を避け、目的に戻らなければなりません: このサービスは本当にビジネスの達成に役立つのか?

3. チームの認知の違い

- フロントエンドエンジニアは迅速な反復を望んでいます。
- バックエンドエンジニアはデータの一貫性を望んでいます。
- 運用エンジニアは高可用性を望んでいます。

アーキテクトは、これらの引っ張り合いの中で「目的志向のバランスポイント」を見つけることができなければなりません。

## 哲学とエンジニアリングの交差点: アリストテレスからクラウドネイティブまで

古代ギリシャの哲学者アリストテレスは「目的因(Final Cause)」の概念を提唱しました: 物事の存在理由は、それが何で作られているかではなく、何の目的を達成するためかです。

ナイフの目的は切ること。
船の目的は航海すること。
システムの目的は特定のビジネス問題を解決すること。

これは、ソフトウェアアーキテクチャで私たちが問う質問と高度に一致しています。

マイクロサービスアーキテクチャを設計しているとき、自分自身に問うことができます:

**このサービスの目的因は何か?**

それが単に**「みんながマイクロサービスを使っているから」**であれば、それは間違った理由です。

その目的が**「複数の国のユーザーによる同時使用をサポートし、独立して迅速にデプロイできるようにするため」**であれば、それには存在理由があります。

哲学は、「ツール崇拝」から「目的志向」に戻るのを助けてくれます。

## 人的要因: チーム、コミュニケーション、意思決定の哲学

アーキテクチャは単なる技術ではなく、チームの共通言語でもあります。

- コミュニケーションの哲学:

- アーキテクトは、抽象的な技術概念をビジネスが理解できる言語に翻訳しなければなりません。
- 同時に、ビジネス要件をエンジニアが実装できる設計に翻訳しなければなりません。

- 意思決定の哲学:

- 多くの場合、「最善の解決策」はなく、「現時点で最も適した解決策」しかありません。
- 例えば: RDS vs DynamoDB。RDSを選択すると将来拡張ボトルネックに遭遇するかもしれませんが、DynamoDBを選択すると一貫性についてより多くの考慮が必要です。

アーキテクトは「限定合理性」を受け入れ、継続的に反復し続けることができなければなりません。

## 未来展望: AIとポストヒューマン時代におけるアーキテクトの役割

AIが自動的にシステムを生成できるようになったとき、アーキテクトは置き換えられるのでしょうか?

答えはノーです。

AIは自動的にコードを生成できますが、「問題定義」が必要です。

アーキテクトの価値は「問題を定義する」ことと「目的を定義する」ことにあります。

ポストヒューマン時代では、アーキテクトはより「デジタル哲学者」のようであり、以下に答える責任があります:

このシステムは存在すべきか? (私は誰か?)
その境界と責任は何か? (私はどこから来たのか?)
人間社会にどのような影響を与えるか? (私はどこへ行くのか?)
